What is Coral?
                                                                                                                                                                              Coral is a CLI tool that lets you query live data sources using SQL.

  Instead of jumping between dashboards, APIs, and UIs, you write SQL queries and Coral fetches the data for you from connected services like:

  - GitHub (PRs, issues, commits)
  - Slack (messages, channels)
  - Linear (tickets, projects)
  - Datadog (metrics, logs)
  - Sentry (errors, events)
  - ...and custom HTTP APIs you define yourself

  Think of it like having a SQL database, except the "tables" are actually live API endpoints.

  ---
  The Mental Model

  coral sql "SELECT * FROM github.pull_requests WHERE repo = 'my-repo' LIMIT 10"
            ^          ^        ^
            CLI cmd    source   table (= API endpoint)

  - Schema = the service (e.g., github, slack, linear)
  - Table = a dataset from that service (e.g., pull_requests, issues)
  - Table functions = tables that require arguments (like calling an API with params)

  ---
  How to Get Started

  Step 1 — Install Coral CLI

  npm install -g @fleetapp/coral-cli   # or however it's distributed in your env

  Step 2 — Discover what's available

  # See all available tables
  coral sql "SELECT schema_name, table_name, description FROM coral.tables"

  # See table functions (API-style endpoints)
  coral sql "SELECT schema_name, function_name, description FROM coral.table_functions"

  Step 3 — Read the guide for a table before querying it

  coral sql "SELECT guide FROM coral.tables WHERE table_name = 'pull_requests'"

  Step 4 — Check required filters (important!)

  Some tables require WHERE clauses or the query will fail:

  coral sql "SELECT column_name, is_required_filter FROM coral.columns WHERE schema_name = 'github' AND table_name = 'pull_requests'"

  Step 5 — Run your first real query

  coral sql "SELECT title, state, created_at FROM github.pull_requests WHERE repo = 'my-org/my-repo' LIMIT 5"

  ---
  What Coral Actually Is (Concrete Mental Model)

  Forget SQL databases for a moment. Imagine this:                                                                                                                          
  PagerDuty API  ──┐
  Datadog API    ──┤──► CORAL ──► You write ONE SQL query across all of them
  GitHub API     ──┤
  StatusGator API──┘

  Coral sits in the middle. It:
  - Handles auth, pagination, rate limits for each API
  - Maps each API's response into a "table" you can query
  - Lets you JOIN across services — like joining PagerDuty incidents with GitHub PRs in one query

  -- This is the power: one query, 3 live APIs
  SELECT
    i.title AS incident,
    p.title AS recent_deploy,
    m.value AS error_rate
  FROM pagerduty.incidents i
  JOIN github.pull_requests p
    ON p.merged_at BETWEEN i.created_at - INTERVAL '2 hours' AND i.created_at
  JOIN datadog.metrics m
    ON m.metric = 'error.rate'
  WHERE i.urgency = 'high'

  That's it. That's Coral's superpower.

  3. Correlate with Datadog error spikes

    i.title AS incident,
    m.metric,

    m.timestamp
  FROM pagerduty.incidents i
  JOIN datadog.metrics m
    ON m.timestamp BETWEEN i.created_at - INTERVAL '15 minutes' AND i.created_at

  4. Who deployed + who's on-call (accountability chain)
  SELECT
    pr.merged_by AS deployer,
    oc.user_name AS on_call_engineer,

  FROM pagerduty.incidents i

  JOIN github.pull_requests pr
    ON pr.merged_at BETWEEN i.created_at - INTERVAL '2 hours' AND i.created_at
  WHERE i.urgency = 'high'

  5. Full timeline reconstruction (the money query)
  SELECT timestamp, source, event_type, description FROM (
    SELECT created_at AS timestamp, 'PagerDuty' AS source, 'incident' AS event_type, title AS description
    FROM pagerduty.incidents WHERE urgency = 'high'
    UNION ALL
    SELECT merged_at, 'GitHub', 'deploy', title FROM github.pull_requests WHERE merged_at > NOW() - INTERVAL '24 hours'
    UNION ALL
    SELECT started_at, 'StatusGator', 'outage', name FROM statusgator.incidents WHERE started_at > NOW() - INTERVAL '24 hours'
  ) timeline
  ORDER BY timestamp DESC

  ---
  Feature 2: Incident DNA Fingerprinting

  Every incident has a signature — which metrics spiked, which service, what deploy happened. Store these fingerprints. Match new incidents against history.

  Incident #4521 → {
    "service": "payment-api",
    "metrics_spiked": ["error.rate", "p99_latency"],
    "deploy_happened": true,
    "deploy_age_minutes": 23,
    "external_outage": false
  }

  New incident comes in → Agent says:
  "This matches 94% with incident #3891 (April 2024).
   Root cause was: database connection pool exhaustion.
   Fixed by: increasing max_connections + restarting pods.
   Resolution time: 8 minutes."

  -- Build the fingerprint database over time
  SELECT
    i.id,
    i.service_name,
    COUNT(DISTINCT m.metric) as metrics_spiked,
    MAX(pr.merged_at) as last_deploy_before_incident,
    TIMESTAMPDIFF(MINUTE, MAX(pr.merged_at), i.created_at) as deploy_age_minutes,
    COUNT(si.id) as external_outages_at_time
  FROM pagerduty.incidents i
  LEFT JOIN datadog.metrics m ON m.timestamp BETWEEN i.created_at - INTERVAL '5 min' AND i.created_at
  LEFT JOIN github.pull_requests pr ON pr.merged_at < i.created_at
  LEFT JOIN statusgator.incidents si ON si.started_at BETWEEN i.created_at - INTERVAL '30 min' AND i.created_at
  WHERE i.urgency = 'high'
  GROUP BY i.id, i.service_name

  Feature 4: Silent Failure Hunter

  Find incidents that should have fired but didn't. Ghost incidents.

  -- Datadog shows error spike, but PagerDuty never fired
  SELECT
    m.timestamp,
    m.metric,
    m.value,
    m.service_name,
    COUNT(i.id) as pagerduty_incidents_fired
  FROM datadog.metrics m
  LEFT JOIN pagerduty.incidents i
    ON i.service_name = m.service_name
    AND i.created_at BETWEEN m.timestamp - INTERVAL '10 min' AND m.timestamp + INTERVAL '10 min'
  WHERE m.metric = 'error.rate'
    AND m.value > 0.05   -- 5% error rate
    AND i.id IS NULL     -- but no PD incident!
  ORDER BY m.value DESC

  Output: "Your payment service had a 12% error rate for 22 minutes on May 12th. Nobody was paged."

  ---
  Feature 5: On-Call Burnout Detector

  -- Who's getting destroyed by pages?
  SELECT
    u.name,
    COUNT(i.id) as pages_this_month,
    AVG(TIMESTAMPDIFF(MINUTE, i.created_at, i.resolved_at)) as avg_resolution_mins,
    COUNT(CASE WHEN HOUR(i.created_at) BETWEEN 0 AND 6 THEN 1 END) as night_pages
  FROM pagerduty.oncalls oc
  JOIN pagerduty.users u ON u.id = oc.user_id
  JOIN pagerduty.incidents i ON i.service_id = oc.service_id
  WHERE i.created_at > NOW() - INTERVAL '30 days'
  GROUP BY u.name
  ORDER BY pages_this_month DESC

  Output: "Sarah has been paged 47 times this month, 18 of them between midnight and 6am. She is a flight risk."

  ---
  Feature 6: Cascade Failure Predictor

  -- If Service A is down, what else breaks? (from Datadog APM dependency data)
  WITH affected_services AS (
    SELECT DISTINCT downstream_service
    FROM datadog.apm_dependencies
    WHERE upstream_service = 'payment-api'
  )
  SELECT
    a.downstream_service,
    COUNT(i.id) as times_this_caused_incident,
    MAX(i.created_at) as last_cascade_incident
  FROM affected_services a
  LEFT JOIN pagerduty.incidents i ON i.service_name = a.downstream_service
  GROUP BY a.downstream_service
  ORDER BY times_this_caused_incident DESC

  ▎ "If payment-api goes down: checkout-service, order-service, fraud-detection will likely cascade. Historical cascade rate: 73%."
